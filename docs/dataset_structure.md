# Dataset Structure and Index Model

## Scope

This dataset is for public-domain or permissively licensed scans of handwritten
Hebrew documents from the modern Hebrew handwriting period. The seeded source
notes mostly target documents written after 1929 and authors who died before
the current Israeli life-plus-70 cutoff.

The corpus should prefer pages that are useful for handwriting recognition:
letters, diaries, notebooks, drafts, speech notes, receipts, forms, marginalia,
and other page-like handwritten artifacts. Printed pages, typescript-only
documents, signatures-only images, vector teaching samples, and non-Hebrew
documents should either be excluded from the scan-level table or marked as
restricted/rejected source leads.

## Recommended Repository Layout

```text
data/
  index/
    sources.jsonl          # One row per source, collection, item, dataset, or lead.
    entries.jsonl          # One row per individual scan/page/note/letter.
  scans/
    <source_id>/
      <entry_id>.<ext>     # Original scan derivative stored by this repository.
  transcriptions/
    <entry_id>/
      plain.txt            # Optional compatible transcript.
      alto.xml             # Optional line/word-aligned transcript.
      hocr.html            # Optional hOCR transcript.
  derived/
    thumbnails/
    normalized/
    ocr/
docs/
  sources/                 # Raw source research notes.
  dataset_structure.md     # This file.
schemas/
  source.schema.json
  entry.schema.json
scripts/
  validate_indexes.py
```

The `data/index/*.jsonl` files are the canonical catalogs. The scan bytes live
under `data/scans/` once ingestion begins. If the scan corpus grows beyond normal
Git comfort, use Git LFS for binary scan files while keeping `data/index/`
ordinary Git text.

## Serialization Format

Use newline-delimited JSON as the canonical editable serialization:

- one complete JSON object per line,
- UTF-8,
- no comments,
- no trailing commas,
- stable sorted keys when generated by tooling,
- ISO 8601 dates where known,
- `null` for unknown scalar values,
- arrays for multiple languages, URLs, licenses, creators, or evidence snippets.

JSONL is preferred over CSV because each source and scan needs nested rights
evidence, acquisition instructions, provenance, dimensions, checksums, and
quality annotations. JSONL is still easy for agents to append, stream, diff, and
validate. Analytics-oriented flat exports are derived from the JSONL by
`scripts/build_exports.py`:

- `exports/entries.csv` — one row per scan, with `files[role=="original"]`
  flattened into `file_*` columns and `creator_count` / `file_count`
  summary columns (committed).
- `exports/sources.csv` — one row per source row (committed).
- `exports/creators.csv` — one row per `(entry_id, position, creator)`
  pair, so consumers can join on `entry_id` without positional gymnastics
  inside a single cell (committed).
- `dist/entries.parquet` — same shape as `entries.csv` with preserved
  types (nullable booleans, `int64` file sizes). Produced under `dist/`
  as an uncommitted build artefact, not under `exports/`, because it is
  binary and large enough to be unfriendly to git diffs.

A SQLite catalog (`exports/catalog.sqlite` or `dist/catalog.sqlite`) is
not built today; add one if and when downstream consumers ask for offline
SQL search. Exports are derived artefacts — the JSONL is still the source
of truth.

## Source Index

`data/index/sources.jsonl` is the mother table of all known source leads. A
source row may describe an institution, collection, item-level archival record,
public dataset, or aggregator category. It does not imply that every page under
that source is eligible.

Required core fields:

- `source_id`: stable lowercase identifier.
- `record_type`: `institution`, `collection`, `item`, `dataset`, or `category`.
- `status`: `candidate`, `verified`, `rejected`, or `needs_review`.
- `priority`: `seed`, `high`, `medium`, `low`, or `exclude`.
- `provider`: institution or platform name.
- `title`: human-readable title.
- `urls`: canonical and supporting URLs.
- `rights`: normalized rights assessment.
- `scope`: expected content scope.
- `ingest`: acquisition and processing hints.
- `evidence`: pointers back to source notes or web evidence.

Rights should be represented conservatively:

- `rights_basis`: `public_domain`, `cc0`, `cc_by`, `cc_by_sa`,
  `restricted`, `unknown`, or `mixed`.
- `license_expression`: SPDX-style expression where possible, such as
  `CC0-1.0`, `CC-BY-4.0`, or `LicenseRef-Public-Domain-Israel`.
- `commercial_use_allowed`: boolean or `null` if not verified.
- `derivatives_allowed`: boolean or `null` if not verified.
- `scan_redistribution_allowed`: boolean or `null` if not verified.
- `attribution_required`: boolean or `null`.
- `evidence_text`: short phrase from the source page or source note.
- `verified_at`: date when rights were checked against the primary source.

Rows with `verification_status` set to `source_note_only`, `unverified`, or
`conflicting` must not set positive permission fields to `true`. Use `null` for
unknown permissions until the primary source page has been checked. Negative
permission fields may still be set to `false` when a source note identifies a
restriction, because that is a conservative exclusion signal.

## Entry Index

`data/index/entries.jsonl` is the second huge table. It should contain one row
per individual scan unit. A multi-page diary becomes many rows. A single image
with two manuscript pages can either be one row with `physical_unit_count: 2` or
two rows if the image is split into page crops.

Required core fields:

- `entry_id`: stable scan-level identifier.
- `source_id`: foreign key into `sources.jsonl`.
- `source_record_id`: provider-side item or file identifier when available.
- `sequence`: page/order data inside the source item.
- `title`: item or page title.
- `creators`: known author/writer/recipient entities.
- `dates`: document creation date and scan access date.
- `languages`: BCP-47 language tags such as `he`, `yi`, `hu`, or `en`.
- `script`: usually `Hebr`, with `Latn` for transliteration cases.
- `document_type`: `letter`, `diary`, `notebook`, `draft`, `speech`,
  `receipt`, `form`, `marginalia`, `postcard`, `poem`, `other`.
- `handwriting`: whether the page is handwritten, mixed, signature-only, or not
  handwriting.
- `files`: local path, original URL, checksum, MIME type, size, dimensions.
- `rights`: scan-level license and rights evidence.
- `provenance`: acquisition timestamp, tool, source URL, and operator/agent.
- `quality`: legibility, crop, skew, blur, color mode, and exclusion reasons.
- `transcription`: optional status, paths, source, producer, and transcript
  rights for text/line/word ground truth.

The entry-level rights record must not simply copy source-level rights blindly.
It should record whether the specific scan page is cleared for redistribution,
commercial use, and derivatives.

Entry rights use the same verification rule as source rights: positive
permission fields may be set to `true` only after primary-page evidence has been
checked for that scan.

Transcription rights are recorded separately from scan rights because a provider
transcript, OCR output, or repo-authored correction can carry different legal
terms than the image itself. If a transcript is available, store compatible
artifacts under `data/transcriptions/<entry_id>/` and set one or more of
`text_path`, `alto_path`, or `hocr_path`. A transcript record must also include
`source_url`, `created_by`, and its own rights object. Positive transcript
permission fields may be set to `true` only after transcript rights evidence has
been checked, or when the transcript is repo-authored metadata released under
the repository metadata license.

## Stable IDs

Use deterministic IDs that survive reharvesting:

```text
source_id = <provider_slug>__<collection_or_item_slug>
entry_id  = <source_id>__p<zero_padded_sequence>
```

For source records with stable provider IDs, include the provider ID in the
slug:

```text
nli__nnl_archive_al997009912248505171
nli__nnl_archive_al997009912248505171__p0001
```

If a provider exposes a IIIF canvas or file ID, use that in `source_record_id`
or `files[].provider_file_id`; do not bury it only in a URL.

## License Policy

The safest repository-level policy is:

- dataset metadata authored in this repository: `CC0-1.0`,
- scans: per-entry license/rights only,
- generated derivatives: inherit the entry's most restrictive applicable terms,
- public release bundles: include only entries where commercial use,
  redistribution, and derivatives are all allowed.

If the corpus mixes public domain, CC0, CC BY, and CC BY-SA scans, the dataset
should be distributed with a compound license notice rather than pretending
there is one uniform scan license. Filtering by `rights.license_expression`
and boolean permission fields should be a first-class workflow.

CC BY-SA-4.0 entries are accepted alongside public-domain, CC0, and CC BY
entries because ShareAlike permits commercial use, redistribution, and
derivatives. Downstream re-distributors of an *adaptation* of a CC BY-SA scan
must release the adaptation under CC BY-SA-4.0 (or a compatible later version).
A release bundle that only aggregates scans without modifying them is not an
adaptation, so the bundle itself does not need to be CC BY-SA-licensed; the
per-entry license metadata travels with the file.

Do not include CC BY-NC, CC BY-NC-SA, CC BY-ND, research-only,
building-access-only, permission-required, or unknown-rights scans in
remix-friendly release bundles.

## Seed Source Assessment

The current source notes suggest these practical buckets:

- strongest seed: National Library of Israel item records around Hannah Senesh
  with `Any Use Permitted` / public-domain evidence,
- promising but item-check needed: NLI/Ktiv, Tchernichovsky, Szold, Jabotinsky
  Institute, Harvard Judaica/HOLLIS, Wikimedia Commons categories,
- permissive but attribution-bearing: TAU/Figshare modern Hebrew baseline,
- exclude from remix-friendly corpus unless relicensed: HHD age/gender/v0
  variants with non-commercial, no-derivatives, or research-only terms,
- reject as non-Hebrew or non-scan if discovered through broad aggregators.

## Ingestion Flow

1. Add or update a `sources.jsonl` row as `candidate`.
2. Verify the source page, rights label, author death date, date written, and
   download/access method.
3. Promote the source row to `verified` only when enough evidence exists.
4. Download the scan into `data/scans/<source_id>/`.
5. Compute checksum, size, MIME type, and dimensions.
6. Add one `entries.jsonl` row per scan unit.
7. Run `python scripts/validate_indexes.py`.
8. Generate derived exports from JSONL only after validation passes.
